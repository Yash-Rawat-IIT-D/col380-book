\chapter{Performance Metrics and Basic PRAM Algorithms}

\section{Metrics We Will Use}
For an input of size $n$, let $T(n,1)$ denote the best sequential running time
and $T(n,p)$ the running time on $p$ processors. The central performance measures
used throughout the lecture are:
\begin{align*}
S(n,p) &= \frac{T(n,1)}{T(n,p)}, \\
C(n,p) &= p \cdot T(n,p), \\
E(n,p) &= \frac{S(n,p)}{p} = \frac{T(n,1)}{C(n,p)}.
\end{align*}
Here $S(n,p)$ is the speedup, $C(n,p)$ is the cost, and $E(n,p)$ is the efficiency.
The parallelism overhead is
\[
T_o(n,p) = C(n,p) - T(n,1).
\]
An algorithm is \ppterm{cost-optimal} when $C(n,p) = O(T(n,1))$, and is
\ppterm{work-efficient} when the total number of primitive operations remains
within a constant factor of the best sequential algorithm. These notions will
be used repeatedly when comparing search, reduction, and prefix-sum algorithms.

\begin{ppkey}
 Fixed-size speedup is limited by the sequential fraction of the computation,
but scalability improves when the problem size grows with the number of processors.
\end{ppkey}

\section{The PRAM Abstraction}
The lecture uses the PRAM model as an idealised machine for algorithm design:
All Processors are synchronous, each processor has a private identity and local
memory, and shared-memory accesses are treated as constant-time operations.
Although unrealistic as a hardware model, it is extremely useful for asymptotic
reasoning and analysis.

\begin{ppdefn}
\ppterm{EREW} (Exclusive Read, Exclusive Write) allows at most one processor to
read or write a shared location in a step. \ppterm{CREW} permits Concurrent
Reads but still requires Exclusive Writes. \ppterm{CRCW} permits both Concurrent
Reads and Concurrent Writes, but with a write-conflict rule for tie breaking.
\end{ppdefn}

Under CRCW, common variants include \ppterm{Priority Write}, where the highest-priority
processor wins; \ppterm{Common Write}, where all concurrent writers must write the
same value; and \ppterm{Arbitrary Write}, where one of the proposed values is chosen
arbitrarily. The model you choose matters because it decides what kinds of 
simultaneous access are allowed, and therefore how much extra coordination the processors need.


\section{Parallel Linear Search}
Suppose an unsorted array $A[1 \dots n]$ is searched for a value $x$ using $p$
processors. A natural block-partitioned strategy assigns roughly $n/p$ elements
to each processor. Each processor performs a local linear scan on its own block
and produces one Boolean answer: whether $x$ was found in that block.

The local scan takes $\Theta(n/p)$ time. These local answers can then be combined
using a tree reduction with logical OR, which takes $\Theta(\log p)$ time. Hence
\[
T(n,p) = \Theta\left(\frac{n}{p} + \log p\right).
\]
The corresponding cost is
\[
C(n,p) = p \cdot T(n,p) = \Theta(n + p \log p).
\]
Therefore the algorithm is cost-optimal when $p \log p = O(n)$; in the usual
case where $n \gg p$, the scan term dominates and this reduction-based 
\hyperref[prog:block-partitioned-search]{solution} is
close to work-efficient.

\begin{ppnote}
The \hyperref[fig:block-search-part]{tree-reduction} version is EREW-safe because processors read disjoint input
blocks and later reduce flags through disjoint pairs. No concurrent access to
the same shared location is required.
\end{ppnote}

The lecture also discussed a stronger shared-memory variant in which processors
write to a shared Boolean register such as \ppinline{found}. In a common-CRCW
model, all successful processors may write the value \ppinline{true}, so the final
combine can be reduced to constant time after the scan phase. This is faster,
but it relies on a stronger concurrency model than the reduction-based EREW version.

An alternate EREW-friendly strategy is to give one element to each processor and
write hits into a private \ppinline{hit[i]} location. A subsequent tree OR-reduction
over the hit array decides whether any processor found the target. With $n$ processors,
this gives $\Theta(\log n)$ time and cleanly separates the private-write phase from
the reduction phase.

\begin{ppprogram}{Parallel linear search with one processor per element}{prog:parallel-linear-search}
\begin{lstlisting}[language=Pseudocode,style=pppseudocode]
Input:  array $A[1 \dots n]$, target value $x$
Output: $\textsc{true}$ iff some entry equals $x$

for all $i \in \{1,\dots,n\}$ in parallel do
    if $A[i] = x$ then
        $R[i] \gets \textsc{true}$ // R[i] is same as hit[i]
    else
        $R[i] \gets \textsc{false}$ 

for $d \gets 0$ to $\lceil \log_2 n \rceil - 1$ do
    for all $i \in \{1,\dots,n\}$ in parallel do
        if $(i - 1) \bmod 2^{d+1} = 0$ and $i + 2^d \le n$ then
            $R[i] \gets R[i] \lor R[i + 2^d]$

return $R[1]$
\end{lstlisting}
\end{ppprogram}

\begin{ppprogram}{Block-partitioned linear search with OR-reduction}{prog:block-partitioned-search}
\begin{lstlisting}[language=Pseudocode,style=pppseudocode]
Input:  array $A[1 \dots n]$, target value $x$, $p$ processors
Output: $\textsc{true}$ iff some entry equals $x$

for all $q \in \{1,\dots,p\}$ in parallel do
    $L_q \gets \left\lfloor \frac{(q - 1)n}{p} \right\rfloor + 1$
    $R_q \gets \left\lfloor \frac{qn}{p} \right\rfloor$
    $F[q] \gets \textsc{false}$
    for $i \gets L_q$ to $R_q$ do
        if $A[i] = x$ then
            $F[q] \gets \textsc{true}$

for $d \gets 0$ to $\lceil \log_2 p \rceil - 1$ do
    for all $q \in \{1,\dots,p\}$ in parallel do
        if $(q - 1) \bmod 2^{d+1} = 0$ and $q + 2^d \le p$ then
            $F[q] \gets F[q] \lor F[q + 2^d]$

return $F[1]$
\end{lstlisting}
\end{ppprogram}

\section{Tree-Based Parallel Addition}
A naive parallelisation of summation would let all processors update one shared
\ppinline{sum} variable. This is a poor design on account of race conditions. 
Thus the updates must be serialised or protected, so the resulting naive
algorithm gains little or no speedup.

One correct approach is a\hyperref[fig:og-tree-sum]{reduction tree}. Assume for simplicity that $n$ is a power
of two. In round 1, adjacent pairs are added. In round 2, the stride doubles and
the partial sums of length two are combined into sums of length four. After
$\log_2 n$ rounds, one processor holds the total sum of all elements.

Counting only the processors that are active in each round, the parallel time is
$\Theta(\log n)$. The total amount of arithmetic work is
\[
\frac{n}{2} + \frac{n}{4} + \cdots + 1 = \Theta(n),
\]
which matches sequential summation up to constant factors. Thus the reduction tree
is work-efficient. Equivalently, if processors that become inactive are released
after each round, the total processor-time cost is also $\Theta(n)$. The larger
$\Theta(n \log n)$ cost appears only if all $n$ processors are charged for all
$\log n$ rounds even after most of them have become idle.

For $p < n$, a standard cost-optimal version first computes local sums over blocks
of size about $n/p$, and then reduces the $p$ block sums in $\Theta(\log p)$ more
steps. This yields
\[
T(n,p) = \Theta\left(\frac{n}{p} + \log p\right),
\]
which matches the block-partitioned search bound. The same two-phase idea is
shown \hyperref[fig:block-tree-sum]{pictorially} and written explicitly in
\hyperref[prog:block-parallel-sum]{the blocked tree-sum pseudocode}.

\begin{pppitfall}
Work and cost differ here only because of how idle processors are charged. The
reduction tree itself performs $\Theta(n)$ additions and is work-efficient. If
inactive processors are freed after each round, the cost stays $\Theta(n)$; if
all $n$ processors are charged across all $\log n$ rounds, the cost becomes
$\Theta(n \log n)$.
\end{pppitfall}

\begin{ppprogram}{Stride-based tree reduction for summation}{prog:parallel-sum}
\begin{lstlisting}[language=Pseudocode,style=pppseudocode]
Input:  array $A[1 \dots n]$, where $n$ is a power of two
Output: total sum stored in $A[n]$

for $d \gets 0$ to $\log_2 n - 1$ do
    for all $k \in \{1,\dots,n\}$ in parallel do
        if $k \bmod 2^{d+1} = 0$ then
            $A[k] \gets A[k - 2^d] + A[k]$

return $A[n]$
\end{lstlisting}
\end{ppprogram}


\begin{ppprogram}{Blocked tree reduction for cost-optimal summation}{prog:block-parallel-sum}
\begin{lstlisting}[language=Pseudocode,style=pppseudocode]
Input:  array $A[1 \dots n]$, $p$ processors
Output: total sum of all entries

for all $q \in \{1,\dots,p\}$ in parallel do
    $L_q \gets \left\lfloor \frac{(q - 1)n}{p} \right\rfloor + 1$
    $R_q \gets \left\lfloor \frac{qn}{p} \right\rfloor$
    $S[q] \gets 0$
    for $i \gets L_q$ to $R_q$ do
        $S[q] \gets S[q] + A[i]$

for $d \gets 0$ to $\lceil \log_2 p \rceil - 1$ do
    for all $q \in \{1,\dots,p\}$ in parallel do
        if $(q - 1) \bmod 2^{d+1} = 0$ and $q + 2^d \le p$ then
            $S[q] \gets S[q] + S[q + 2^d]$

return $S[1]$
\end{lstlisting}
\end{ppprogram}

\section{Parallel Prefix Sums}
The prefix-sum or scan problem asks us to transform an input array
$A[1 \dots n]$ into prefix sums. In the \ppterm{inclusive} version,
\[
B[i] = \sum_{j=1}^{i} A[j],
\]
whereas in the \ppterm{exclusive} version,
\[
B[i] = \sum_{j=1}^{i-1} A[j],
\]
with the convention that $B[1] = 0$. Sequentially, either version takes $\Theta(n)$
time.

\subsection{Hillis-Steele Scan}
The \hyperref[prog:hillis-steele]{Hillis-Steele} scan proceeds in rounds. At round $d$, processor $i$ adds into
its current value the entry from position $i - 2^{d-1}$, provided that this index
exists. Conceptually, each round must read values from the \emph{previous} round,
not partially updated values from the current round. For that reason, the
pseudocode below uses a temporary array to hold the next-round values before
copying them back.

\begin{ppprogram}{Hillis-Steele inclusive scan}{prog:hillis-steele}
\begin{lstlisting}[language=Pseudocode,style=pppseudocode]
Input:  array $A[1 \dots n]$
Output: inclusive prefix sums $B[1 \dots n]$

for all $i \in \{1,\dots,n\}$ in parallel do
    $B[i] \gets A[i]$

for $d \gets 1$ to $\lceil \log_2 n \rceil$ do
    for all $i \in \{1,\dots,n\}$ in parallel do
        if $i > 2^{d-1}$ then
            $\mathrm{Next}[i] \gets B[i] + B[i - 2^{d-1}]$
        else
            $\mathrm{Next}[i] \gets B[i]$

    for all $i \in \{1,\dots,n\}$ in parallel do  
        // Effectively O(1) time in our model
        $B[i] \gets \mathrm{Next}[i]$

return $B$
\end{lstlisting}
\end{ppprogram}

The temporary array is essential in this formulation: if processors overwrote
$B[i]$ immediately, then later reads in the same round could observe a mixture of
old and new values. The second parallel loop is the commit step that turns the
newly computed values into the state for the next round.

\paragraph{Correctness proof by induction.}
Let $B^{(0)}[i] = A[i]$. For each round $d \ge 1$, let $B^{(d)}$ denote the
contents of array $B$ \emph{after} the copy-back step at the end of round $d$.
Then
\[
B^{(d)}[i] =
\begin{cases}
B^{(d-1)}[i] + B^{(d-1)}[i - 2^{d-1}], & \text{if } i > 2^{d-1}, \\
B^{(d-1)}[i], & \text{otherwise}.
\end{cases}
\]
We claim that after round $d$, every index $i$ stores the sum of the last
$\min(2^d,i)$ input values ending at position $i$:
\[
B^{(d)}[i] = \sum_{j=\max(1,\, i - 2^d + 1)}^{i} A[j].
\]
The proof is by induction on $d$.

For the base case $d = 0$, we have $B^{(0)}[i] = A[i]$, which is exactly the sum
over the last $\min(1,i) = 1$ entry ending at $i$.

Now assume the claim holds after round $d-1$. Fix an index $i$.
If $i \le 2^{d-1}$, then no update happens in round $d$, so
$B^{(d)}[i] = B^{(d-1)}[i]$. By the induction hypothesis this is already
\[
\sum_{j=1}^{i} A[j]
= \sum_{j=\max(1,\, i - 2^d + 1)}^{i} A[j],
\]
because $i \le 2^{d-1}$ implies both $i - 2^{d-1} + 1 \le 1$ and
$i - 2^d + 1 \le 1$.

If instead $i > 2^{d-1}$, then by the update rule and the induction hypothesis,
\begin{align*}
B^{(d)}[i]
&= B^{(d-1)}[i] + B^{(d-1)}[i - 2^{d-1}] \\
&= \sum_{j=i-2^{d-1}+1}^{i} A[j]
 + \sum_{j=\max(1,\, i-2^d+1)}^{i-2^{d-1}} A[j] \\
&= \sum_{j=\max(1,\, i-2^d+1)}^{i} A[j].
\end{align*}
The two sums are over adjacent ranges, so together they cover exactly the last
$\min(2^d,i)$ entries ending at position $i$.

Therefore the claim holds for every round $d$. After
$D = \lceil \log_2 n \rceil$ rounds we have $2^D \ge n$, so for every $i$,
\[
B^{(D)}[i] = \sum_{j=1}^{i} A[j],
\]
which is precisely the inclusive prefix sum. Hence the Hillis-Steele algorithm
is correct.

For the example array
\[
[3,1,4,5,1,9,2],
\]
the inclusive scan evolves as follows:
% \[
% [3,4,5,9,6,10,11],
% \]
% \[
% [3,4,8,13,11,19,17],
% \]
% \[
% [3,4,8,13,14,23,25].
% \]

After $\lceil \log_2 n \rceil$ rounds, every position holds the correct prefix sum.

The time is $\Theta(\log n)$ on $n$ processors, but every round touches $\Theta(n)$
entries, so the total work is $\Theta(n \log n)$. Thus Hillis-Steele is easy to
parallelise, but it is not work-efficient.

\subsection{Blelloch Scan}
\hyperref[prog:blelloch]{Blelloch's algorithm} improves the work bound by using two tree phases. 
For simplicity, assume the input size is a
power of two; otherwise pad the array with zeros up to the next power of two.

In the \ppterm{upsweep} (or reduction) phase, a balanced binary tree is built over
the array. Each internal node stores the sum of its two children, so the root
eventually stores the total sum of all elements. This phase resembles tree-based
parallel addition.

In the \ppterm{downsweep} phase, the root is first set to $0$. Then each internal
node propagates prefix information downward as follows. If the parent currently
stores $p$ and the left child previously stored a subtotal $t$, then the left child
receives $p$ and the right child receives $p + t$. Repeating this from the root
to the leaves yields the \emph{exclusive} scan.

The inclusive scan can be recovered by adding the original input element-wise to
the exclusive result. As shown \hyperref[fig:blelloch-scan]{here}, the upsweep
phase is represented by purple arrows and the downsweep phase by green arrows.
At level $d$ of either phase, only $\frac{n}{2^{d+1}}$ internal updates are
performed, so each phase does
\[
\frac{n}{2} + \frac{n}{4} + \cdots + 1 = O(n)
\]
work. Since the green and purple phases differ only by a constant factor of two,
the total work remains $O(n)$, while the parallel time is still $\Theta(\log n)$.
Therefore Blelloch scan is work-efficient.

\begin{ppprogram}{Blelloch exclusive scan with inclusive conversion}{prog:blelloch}
\begin{lstlisting}[language=Pseudocode,style=pppseudocode]
Input:  array $A[0 \dots n-1]$, where $n$ is a power of two
Output: exclusive scan $B[0 \dots n-1]$

for all $i \in \{0,\dots,n-1\}$ in parallel do
    $B[i] \gets A[i]$

// Upsweep phase
for $d \gets 0$ to $\log_2 n - 1$ do
    for all $k \in \{0, 2^{d+1}, 2 \cdot 2^{d+1}, \dots, n - 2^{d+1}\}$ in parallel do
        $B[k + 2^{d+1} - 1] \gets B[k + 2^d - 1] + B[k + 2^{d+1} - 1]$

$B[n-1] \gets 0$

// Downsweep phase
for $d \gets \log_2 n - 1$ downto $0$ do
    for all $k \in \{0, 2^{d+1}, 2 \cdot 2^{d+1}, \dots, n - 2^{d+1}\}$ in parallel do
        $t \gets B[k + 2^d - 1]$
        $B[k + 2^d - 1] \gets B[k + 2^{d+1} - 1]$
        $B[k + 2^{d+1} - 1] \gets t + B[k + 2^{d+1} - 1]$

return $B$
\end{lstlisting}
\end{ppprogram}

\begin{ppkey}
Hillis-Steele gives a simple scan in $\Theta(\log n)$ time but spends $\Theta(n \log n)$
work. Blelloch keeps the same asymptotic time while reducing the work to $\Theta(n)$.
\end{ppkey}

For block-partitioned search, the work bound is $\Theta(n + p \log p)$, so the
algorithm is work-efficient when $p \log p = O(n)$.

\begin{itemize}
\item The PRAM model separates algorithm design from machine details, but the choice of EREW, CREW, or CRCW still changes which synchronisation patterns are legal.
\item Tree-structured reductions give the core design pattern behind both parallel search and parallel summation.
\item Hillis-Steele achieves logarithmic parallel time, but it pays for that simplicity with $\Theta(n \log n)$ total work.
\item Blelloch retains logarithmic parallel time while reducing the work to $\Theta(n)$, making it the asymptotically stronger scan algorithm.
\item Speedup alone can be misleading; work, cost, and work-efficiency are necessary to judge whether a parallel algorithm is truly worthwhile.
\end{itemize}
